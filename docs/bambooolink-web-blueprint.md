# BambooLink 网页结构蓝图（面向 AI 协同开发）

> 角色视角：资深 B 端产品经理 + AI 协同开发负责人  
> 目标：把产品方案进一步转化为可直接交给 AI 编码工具落地的 Web 产品蓝图

---

## 1. 网站整体结构方案

### 1.1 顶部导航设计（登录前）

**导航结构（左 Logo + 右导航 + 行动按钮）**
- 左侧：`BambooLink`（点击回首页）
- 中间导航：平台机制 / 角色入口 / 规则与可信 / 平台公告
- 右侧按钮：`登录`、`注册`、`进入工作台`（已登录时替换为头像菜单）

**设计原则**
- 导航文案强调“协作与机制”，避免“立即购买”类消费电商语义。
- 进入工作台按钮为低调主按钮，不使用高饱和营销色。

### 1.2 首页定位：展示页 + 业务入口并存

**结论：首页既是展示页，也是业务入口页。**
- 对未登录用户：解释 BambooLink 的产业价值与流程机制。
- 对已登录用户：顶部保留平台说明，下方优先展示“按角色进入工作台”。

**原因**
- 正式网站需要“可理解”，必须有完整业务叙事首页。
- 业务使用需要“可执行”，必须快速进入工作台。

### 1.3 登录前一级页面

1. 首页（平台价值与流程）
2. 登录页
3. 注册与角色选择页
4. 平台规则公开页（审核、履约、争议规则）
5. 平台公告与更新页（可选）

### 1.4 登录后一级页面（通用骨架）

1. 工作台
2. 资源中心
3. 询价协同
4. 订单与履约
5. 争议处理
6. 信用档案
7. 规则中心
8. AI 匹配（占位）

### 1.5 角色化导航差异

#### 竹农导航
- 工作台
- 我的资源（发布/审核状态）
- 收到的询价
- 履约任务
- 我的信用
- 规则中心

#### 竹企导航
- 工作台
- 资源市场（筛选/收藏/对比）
- 我的询价
- 我的订单
- 供应方信用
- 规则中心

#### 管理员导航
- 管理总览
- 资源审核
- 用户与资质
- 争议工单
- 治理指标
- 规则配置
- AI 匹配管理（占位）


### 1.6 迭代与提交基线

- 所有后续迭代默认基于 GitHub `main` branch。
- 直接在 `main` 提交，不创建 feature/dev 分支。
- 页面与组件演进均以 `main` 当前代码为唯一基线。

## 2. 核心页面优先级排序（MVP）

> 优先级标准：是否直接支撑“快速匹配与互信达成”主链路

### P0（首批必须完成）

1. **登录/注册页**
   - 原因：没有身份体系就无法进入角色化流程。
2. **角色工作台**
   - 原因：所有任务入口与待办聚合都在此，决定可用性。
3. **资源列表页**
   - 原因：供需匹配起点，决定检索效率。
4. **资源详情页**
   - 原因：交易判断核心页面，必须承载“可信信息”。
5. **资源发布页（竹农）**
   - 原因：供给侧冷启动关键，影响资源池质量。
6. **询价会话页**
   - 原因：报价协商与版本留存是平台差异化核心。
7. **订单详情页**
   - 原因：状态流转与履约协作中心。
8. **管理后台首页**
   - 原因：审核与治理能力是平台可信基础。

### P1（第二批）

9. 首页（精细化展示版）
   - 原因：P0 可先有基础入口；第二批完善叙事和展示细节。
10. 争议处理页
11. 信用档案页
12. 规则中心页
13. AI 匹配占位页

### P2（第三批优化）
- 对比分析页
- 治理报表页
- 操作日志与审计页
- 运营活动与公告页

## 4. 核心页面模块级拆解

> 格式：页面名称 - 页面目标 - 页面模块 - 模块内容 - 关键交互

### 4.1 登录/注册页
- **页面目标**：完成身份进入与角色绑定。
- **页面模块**：登录表单 / 注册表单 / 角色选择。
- **模块内容**：手机号或邮箱、验证码/密码、企业或农户身份信息。
- **关键交互**：选择角色后展示对应权限说明；注册后引导进入首次设置。

### 4.2 角色工作台
- **页面目标**：汇总待办，驱动用户执行下一步。
- **页面模块**：待办卡片、流程看板、风险提醒、快捷入口。
- **模块内容**：待回复询价、待确认订单、逾期节点、近期争议。
- **关键交互**：点击待办直达对应详情；支持按紧急程度排序。

### 4.3 资源列表页
- **页面目标**：提高供需检索与筛选效率。
- **页面模块**：筛选器、结果列表、排序条、对比栏。
- **模块内容**：产地、规格、库存、可交付时间、信誉等级。
- **关键交互**：多条件筛选即时刷新；勾选多条资源后进入对比视图。

### 4.4 资源详情页
- **页面目标**：支撑“是否发起询价”的判断。
- **页面模块**：资源摘要、交易条件、可信档案、操作区。
- **模块内容**：规格参数、起订量、报价区间、历史履约率、争议摘要。
- **关键交互**：发起询价前需确认关键条款；支持收藏与分享。

### 4.5 资源发布页（竹农）
- **页面目标**：高质量发布资源并进入审核。
- **页面模块**：基础信息表单、附件上传、草稿区、提交区。
- **模块内容**：资源批次、等级、含水率、可供周期、检测报告。
- **关键交互**：字段校验+缺失提示；支持“保存草稿”“提交审核”。

### 4.6 询价会话页
- **页面目标**：沉淀可追溯的报价协商记录。
- **页面模块**：会话时间线、版本对比、条款确认、备注区。
- **模块内容**：每轮报价、数量、交付日期、验收标准变化。
- **关键交互**：选择两个版本进行差异对比；确认后生成订单草案。

### 4.7 订单详情页
- **页面目标**：统一管理交易状态与履约动作。
- **页面模块**：状态流转、条款快照、履约节点、凭证附件。
- **模块内容**：订单状态机、双方确认记录、运输/验收凭证。
- **关键交互**：节点完成后上传凭证并触发对方确认；异常可一键发起争议。

### 4.8 管理后台首页
- **页面目标**：集中治理与风险监控。
- **页面模块**：治理指标看板、审核队列、争议队列、风险主体列表。
- **模块内容**：审核时效、争议关闭时长、高风险账户、异常趋势。
- **关键交互**：从指标下钻到具体工单；支持按风险等级筛选。

### 4.9 首页（对外）
- **页面目标**：建立平台认知并引导进入业务流程。
- **页面模块**：Hero、价值区、流程区、角色入口、可信机制、AI 预留。
- **模块内容**：机制叙事、关键指标、角色入口按钮。
- **关键交互**：点击“进入工作台/查看机制”跳转对应模块。

## 5. 页面文案语气建议

### 5.1 整体语气基调
- 专业、克制、明确。
- 结论先行，避免口号堆砌。
- 使用“可执行动作 + 可验证结果”的表达。

### 5.2 首页标题语气
- 推荐：`描述问题 + 给出机制解法`
- 避免：`颠覆行业、重塑未来`等夸张营销词。

### 5.3 按钮文案风格
- 用动词开头、短语义：
  - `发起询价`
  - `提交审核`
  - `确认条款`
  - `上传凭证`
- 避免：`立即抢购`、`马上下单`等 C 端导向文案。

### 5.4 后台状态文案风格
- 采用“状态 + 条件”结构：
  - `待审核（资料已提交）`
  - `履约中（待上传运输凭证）`
  - `异常预警（节点逾期 24 小时）`

### 5.5 提示语与异常提醒风格
- 先事实、后建议：
  - `当前订单验收凭证缺失，建议在 24 小时内补充上传。`
- 避免情绪化和指责性语句。

## 6. 前端实现建议（AI 协同开发导向）

### 6.1 推荐前端技术栈

**基础建议（优先“快搭建+好维护”）**
- 框架：Next.js（App Router）或 React + Vite
- 语言：TypeScript
- UI：Tailwind CSS + 组件库（如 shadcn/ui 或 Ant Design）
- 状态管理：Zustand（轻量）
- 数据请求：TanStack Query（后续对接接口时平滑迁移）

### 6.2 页面组织方式

```
/app
  /(public)      # 登录前页面：首页、登录、注册、规则
  /(farmer)      # 竹农端
  /(enterprise)  # 竹企端
  /(admin)       # 管理后台
/components
  /layout
  /cards
  /tables
  /forms
  /timeline
/modules
  /resource
  /inquiry
  /order
  /governance
```

### 6.3 组件拆分思路
- 全局组件：TopNav、SideNav、StatusBadge、MetricCard
- 业务组件：ResourceCard、QuoteTimeline、VersionDiffPanel、OrderStepper
- 复合组件：TrustProfilePanel、DisputeEvidenceList、RiskAlertBanner
- 原则：先做“可复用信息容器”，再做页面拼装。

### 6.4 适合先用 Mock Data 的页面
- 首页（指标与流程卡）
- 资源列表/详情
- 询价会话（版本对比）
- 订单详情（状态流转）
- 管理后台总览

> 理由：这些页面核心是信息结构与交互节奏，早期不依赖真实接口即可验证产品逻辑。

### 6.5 需要预留 AI 匹配模块的位置
- 首页：AI 匹配预告模块
- 资源列表页顶部：`智能推荐资源`
- 询价会话右侧：`相似成交参考`
- 管理后台：`推荐采纳率/命中率`占位卡片

### 6.6 适合 vibe coding 的实现方式
1. 先固定设计 token（颜色、间距、字体、圆角）。
2. 用 AI 先生成 layout skeleton（页面骨架）。
3. 用 mock schema 统一字段（资源、询价版本、订单节点）。
4. 按“列表→详情→协作流程”逐步联动。
5. 每完成一页就补充页面级验收清单（非技术文档也可）。

## 7. 给 AI 编码工具的分阶段拆解策略

### 阶段 1：静态官网与首页框架
- **目标**：建立 BambooLink 的品牌与机制认知。
- **页面范围**：首页、登录页、注册页、规则页（基础版）。
- **数据要求**：不需要真实数据，全部 mock。
- **开发形态**：纯静态优先，强调布局与文案。

### 阶段 2：登录后骨架与资源中心
- **目标**：打通角色化导航与资源主链路入口。
- **页面范围**：角色工作台、资源列表、资源详情、资源发布。
- **数据要求**：mock data + 本地状态管理。
- **开发形态**：静态页面 + 轻交互动态。

### 阶段 3：询价与订单流程
- **目标**：完成“匹配—协商—确认—履约”的核心闭环。
- **页面范围**：询价会话、订单详情、履约节点组件。
- **数据要求**：mock data（含版本历史和状态机）。
- **开发形态**：动态交互优先，接口仍可占位。

### 阶段 4：管理后台与治理能力
- **目标**：补齐平台可信机制与治理闭环。
- **页面范围**：管理后台首页、资源审核、争议队列、信用档案。
- **数据要求**：mock data 为主，可逐步替换真实接口。
- **开发形态**：动态看板 + 队列操作。

### 阶段 5：AI 匹配占位整合
- **目标**：预留后续智能能力入口，完成业务闭环。
- **页面范围**：首页 AI 区、资源推荐区、后台 AI 管理占位。
- **数据要求**：mock 推荐结果 + mock 推荐原因。
- **开发形态**：静态展示 + 可解释标签交互。

## 补充：可直接给 AI 编码工具的执行原则（简版）

1. 所有页面优先复用统一布局与状态标签组件。
2. 所有业务对象先定义前端类型：`Resource`、`InquiryVersion`、`OrderNode`。
3. 所有关键流程必须可视化：版本变化、状态流转、异常提醒。
4. 所有高风险动作需二次确认：提交审核、确认条款、发起争议。
5. 所有 AI 相关功能只做“入口与解释位”，不做算法实现。
