# BambooLink 项目复盘与对外说明材料

## 1. 项目概述

### 1.1 项目背景

BambooLink 的业务语境来自竹产业协作中常见的多主体、跨环节问题：资源信息不透明、询价沟通链路割裂、履约节点难追踪、争议处理标准不一致、治理策略缺乏连续记录。当前版本的核心任务，是先用完整前端系统把这条业务链路搭建清楚，为后续真实工程化接入打基础。

### 1.2 产品定位

BambooLink 定位为面向竹产业协作场景的 Web 协同系统，采用“对外站点 + 登录后工作台”双层结构。对外站点承担规则公开与角色引导，工作台承担业务链路推进与状态留痕。

### 1.3 目标用户

- 竹农主体
- 竹企主体
- 加工协作方
- 物流协作方
- 平台协作/治理角色

### 1.4 解决的核心问题

- 将资源、询价、履约、争议、治理打通为统一链路，减少跨页面信息割裂。
- 用状态标签、筛选、记录表格实现“可查看、可追踪、可回看”的协作表达。
- 在尚未接入后端时，先固化信息架构、字段结构与页面节奏，降低后续联调成本。

### 1.5 选择 Web 而非小程序的原因

- **复杂信息密度更适配 Web**：履约、争议、治理等模块存在多字段表格与筛选组合，Web 更适合承载高密度 B 端信息。
- **跨角色协作与后台式操作更自然**：侧边导航 + 顶栏 + 多模块并行的系统形态在 Web 上认知成本更低。
- **工程扩展性更高**：当前项目已形成可扩展组件体系和路由体系，便于逐步接入权限、审批、审计等能力。

## 2. 核心业务链路总结

当前系统已形成以下前端闭环链路：

1. **资源协同**（`/resource-collab`）
   - 作用：管理资源基础信息、状态与协作入口。
   - 关键能力：筛选、状态识别、详情预览、发起协同入口。
2. **询价报价**（`/inquiry-quotes`）
   - 作用：承接资源协同后的沟通与报价推进。
   - 关键能力：状态概览、角色视角筛选、进入履约入口。
3. **履约跟踪**（`/fulfillment`）
   - 作用：跟踪协作执行阶段与风险状态。
   - 关键能力：阶段状态、风险识别、回看询价、异常进入规则争议。
4. **规则与争议**（`/rules-disputes`）
   - 作用：承接异常与争议事项处理。
   - 关键能力：规则提示、处理原则、关联记录上下文、进入治理上下文。
5. **平台治理**（`/governance`）
   - 作用：汇总主体风险和治理状态，形成平台层观察与治理动作依据。
   - 关键能力：主体视图、风险等级、治理状态、链路数据聚合。

衔接方式采用“链路入口 + 查询参数上下文 + 统一字段语义”的方式实现，即使暂未接后端，也能保持流程完整感。

## 4. 产品设计亮点（复盘视角）

1. **链路驱动的信息架构**：不是孤立页面集合，而是沿“资源→询价→履约→争议→治理”的链路组织功能。
2. **从公开站点到系统工作台的自然过渡**：前台负责认知与规则透明，后台负责执行与留痕，结构职责清晰。
3. **规则先行的协作表达**：规则中心、争议模块、治理模块形成闭环，突出平台治理属性。
4. **多角色协作视角**：在询价、履约、治理中体现发起方/接收方、主体类型与风险视角差异。
5. **状态语义可视化**：各模块统一状态标签体系，支持快速识别阶段、风险与处理进度。
6. **可追踪的上下文串联**：通过模块间入口与参数承接保持业务上下文连续性。
7. **页面节奏统一**：标题区、概览卡、筛选区、表格区、提示区形成一致的系统语言，便于长期扩展。

## 5. 工程与实现亮点

1. **public/workspace 双层路由结构**：在 App Router 下明确区分展示层与系统层，便于团队协作与模块扩展。
2. **设计 token 与样式基线**：全局 token + workspace 样式映射，保证颜色、间距、边框、阴影一致。
3. **auth 页面体系抽象**：登录/注册共享布局与样式基线，避免重复开发。
4. **workspace 通用组件沉淀**：`WorkspacePageHeader`、`FilterBar`、`MetricCard`、多类 Table/StatusTag，可复用性高。
5. **mock 数据驱动开发**：在 `constants/workspace.ts` 中集中管理字段与文案，降低后续接口对接成本。
6. **链路型前端交互预埋**：通过模块间入口和查询参数承接，实现“无后端条件下的流程联通”。
7. **GitHub + Vercel 部署路径明确**：当前工程可直接对接标准前端托管流程，便于交付与评审。

## 7. 面试讲述提纲（3~5 分钟）

### 开场（30 秒）
- 先交代业务背景：竹产业协作存在多主体、跨环节信息断点。
- 再交代目标：我把它拆成“公开入口 + 登录后协作系统”的完整 Web 方案。

### 业务拆解（1~1.5 分钟）
- 说明五段链路：资源协同、询价报价、履约跟踪、规则争议、平台治理。
- 讲每段的输入输出关系：例如询价承接资源，履约承接询价，争议承接异常，治理承接争议聚合。

### 产品设计方法（1 分钟）
- 强调“链路优先”而不是“页面优先”。
- 强调规则和治理能力前置，避免只做交易流程。
- 强调多角色与状态语义，确保系统可解释、可追踪。

### 实现落地（1 分钟）
- 工程上采用 public/workspace 双层结构。
- 用组件抽象和 mock 数据驱动保障可扩展性。
- 页面之间通过统一入口和上下文字段衔接，先把交互链路跑通。

### 收束（30 秒）
- 当前版本价值：前端层已形成完整系统闭环，可用于产品验证和工程接入前准备。
- 下一步重点：接后端、做权限与审批、做真实状态流转与审计。

## 8. 简历补充说明素材（产品经理向）

我在 BambooLink 项目中负责从 0 到 1 的前端产品化落地，核心工作不是单页设计，而是围绕竹产业协作建立完整业务链路。项目采用“公开站点 + 登录后工作台”的双层结构：公开站点承担规则透明与角色引导，工作台承载资源协同、询价报价、履约跟踪、规则争议和平台治理五个连续模块。

在业务设计上，我重点处理了多主体协作中的链路衔接问题，通过统一字段结构、状态语义和模块间入口，形成可追踪、可回看的协作流程。即使在未接入后端的阶段，也确保系统具备可交互的流程感和明确的边界。

在落地层面，我推动了页面结构、文案语气、状态标签、筛选与表格节奏的统一，并以组件化方式沉淀工作台通用能力，保证后续扩展不会反复重构。最终产出是一个可汇报、可讲解、可继续工程化接入的完整前端系统版本。
